  <files>docs/MIGRATIONS.md</files>
  <action>
Create `docs/MIGRATIONS.md` documenting the migration workflow for this project. The document should cover:

**Overview:**
- Migrations live in `supabase/migrations/` with numeric prefix naming (e.g., `00025_notification_types.sql`)
- Currently 24 migration files exist (00001 through 00025, with 00024 skipped)
- Migrations are applied manually via the Supabase SQL Editor or CLI

**Pre-deploy checklist:**
1. Write the migration SQL file in `supabase/migrations/` following the `NNNNN_description.sql` naming convention
2. Test the migration against a local or staging Supabase instance if available (`supabase db reset` for local)
3. Apply the migration to production Supabase via SQL Editor (Dashboard > SQL Editor > paste and run)
4. Verify the migration succeeded (check table structure, run a test query)
5. Deploy the application code that depends on the new schema

**Rollback procedure:**
- Each migration file should have a corresponding rollback strategy documented in a comment at the top of the file
- For additive changes (new tables, columns, indexes): rollback is DROP TABLE/COLUMN/INDEX
- For constraint changes (ALTER TABLE ADD CONSTRAINT): rollback is DROP CONSTRAINT
- For destructive changes (DROP COLUMN, data transforms): rollback requires restoring from backup -- note that Supabase provides point-in-time recovery on Pro plan
- Emergency rollback steps: 1) Write reverse SQL, 2) Apply via SQL Editor, 3) Redeploy previous app version if needed

**Pending migrations (as of project state):**
- `00013_search_indexes.sql` -- full-text search indexes
- `00023_featured_expert.sql` -- featured expert functionality
- `00025_notification_types.sql` -- new notification types

**Best practices:**
- Always apply migrations BEFORE deploying code that depends on them
- Keep migrations small and focused (one concern per file)
- Include IF NOT EXISTS / IF EXISTS guards where possible for idempotency
- Never modify an already-applied migration -- create a new one instead
